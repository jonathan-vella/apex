## 🔍 Context

The RFQ requires EU-only processing, replication, indexing, and caching unless Contoso provides a
written exception. The Step 2 architecture assessment identified a critical sovereignty finding:
Azure Front Door is excluded from the Azure EU Data Boundary, which means WAF, routing metadata,
and edge caching can involve non-EU processing.

The same sovereignty review also raises two related constraints for identity and tenant setup:

- Entra External ID MFA methods that depend on PSTN, SMS, or push notification flows may process
  data outside the EU boundary.
- Azure EU Data Boundary configuration is a tenant prerequisite that must be enabled before the
  target subscriptions are created.

This requires a compliance strategy, not just a service choice.

## ✅ Decision

Adopt an EU Data Boundary baseline with three rules:

1. Treat Application Gateway WAF as the default compliant ingress path for the core platform.
2. Restrict Entra External ID MFA to FIDO2, passkeys, or TOTP-based methods unless Contoso grants
   a written sovereignty exception.
3. Require EU Data Boundary tenant configuration before project subscriptions are created.

Azure Front Door may only be reintroduced through a formally documented exception path with
explicit Contoso approval.

## 🔄 Alternatives Considered

- `EU boundary baseline with Application Gateway, MFA restrictions, and tenant prerequisite`
  Pros: strongest alignment with the RFQ sovereignty requirement and lowest compliance ambiguity.
  Cons: gives up global edge CDN benefits and adds governance gates up front.
  WAF impact: Security `+`, Reliability `→`, Performance `-`, Cost `→`, Operations `-`.
- `Retain Front Door with written exception`
  Pros: preserves anycast acceleration, integrated CDN, and simpler global edge posture.
  Cons: keeps a known compliance deviation in the design and requires a formal exception.
  WAF impact: Performance `+`, Compliance `-`, Security `-`.
- `Allow all Entra MFA methods`
  Pros: simpler user experience and broader MFA channel coverage.
  Cons: leaves a known residual data-boundary gap for PSTN and push-based methods.
  WAF impact: Operations `+`, Compliance `-`.
- `Use a tenant without EU Data Boundary enablement`
  Pros: faster short-term setup if an existing tenant already exists.
  Cons: creates a foundational sovereignty risk that cannot be cleanly corrected later.
  WAF impact: Schedule `+`, Compliance `-`.

## ⚖️ Consequences

### Positive

- Aligns the design to the RFQ's strictest EU-only processing requirement before implementation.
- Makes the compliant ingress baseline explicit by preferring Application Gateway over Front Door.
- Establishes a clear identity control baseline for Entra External ID MFA.
- Forces the tenant prerequisite to be handled early, when it is still cheap to correct.

### Negative

- Gives up Front Door's global CDN and anycast benefits unless an exception is approved.
- Narrows the available MFA experience for end users and administrators.
- Adds governance work and decision gates before subscription creation and production rollout.

### Neutral

- A later business case can still justify Front Door, but only through a documented exception and
  compensating controls.
- The decision does not remove the need to monitor Microsoft's published EU Data Boundary posture
  for identity and edge services.

## 🏛️ WAF Pillar Analysis

- Security: `↑` Reduces sovereignty ambiguity and constrains identity flows to lower-risk MFA
  methods.
- Reliability: `→` Remains acceptable, but regional ingress removes Front Door's global edge
  failover posture.
- Performance: `↓` Regional ingress may not match Front Door for global edge acceleration.
- Cost: `→` Application Gateway is an acceptable compliant alternative without materially changing
  the cost envelope at this stage.
- Operations: `↓` Requires tighter tenant governance, MFA policy management, and exception
  handling.

## 🔒 Compliance Considerations

- Directly addresses the GDPR-aligned EU data residency requirement called out in RFQ §4.3.
- Supports PCI-DSS segmentation by keeping public ingress regional and identity controls explicit.
- Requires governance records for any approved exception involving Front Door or non-EU MFA
  channels.
- Must be paired with regional redundancy choices such as ZRS and private connectivity to the data
  tier.

## 📝 Implementation Notes

- Build the Step 4 planning baseline around Application Gateway WAF, API Management, AKS, and
  private data services in `swedencentral`.
- Configure Entra External ID policies to allow FIDO2, passkeys, and TOTP while disabling SMS and
  push-based MFA unless explicitly approved.
- Confirm EU Data Boundary tenant readiness before provisioning subscriptions, policies, and
  workloads.
- Maintain an explicit architecture exception register for any requested use of Front Door.
