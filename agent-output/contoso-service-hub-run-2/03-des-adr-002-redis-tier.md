## 🔍 Context

GAP-02 required a final cache tier decision for a platform that needs a nominal 128 GB in-memory cache for sessions,
availability reads, and hot-path application data. The architecture assessment compared Azure Managed Redis, Redis
Premium, and Redis Enterprise family options while also accounting for March 2026 pricing and the operational reality
that usable cache memory is lower than raw advertised capacity.

Premium P4 high availability pricing is materially higher than Azure Managed Redis, and its usable capacity remains
below the architecture target. Enterprise E100 is also under the target while costing more. Azure Managed Redis M150
is cost-efficient, but once reserved service memory is accounted for it does not reliably satisfy a 128 GB usable
working set. That leaves Azure Managed Redis M200 as the safest baseline without introducing a clustered workaround.

## ✅ Decision

Use Azure Managed Redis M200 as the default production cache tier for Contoso Service Hub.

M200 is selected because it provides enough headroom to meet the platform’s 128 GB effective cache requirement while
remaining materially cheaper than Premium P4 high availability and Enterprise E100. If Contoso later clarifies that
the RFQ requirement refers to raw provisioned size rather than usable cache memory, M150 can be revisited as a cost
optimization after launch.

## ⚖️ Consequences

### Positive

- Meets the effective cache sizing target without relying on clustered or under-provisioned alternatives.
- Preserves a managed, private-endpoint-capable service model that fits the rest of the architecture.
- Avoids a near-term redesign triggered by cache memory pressure during production adoption.
- Maintains a meaningful cost advantage over Premium P4 and Enterprise E100.

### Negative

- Costs more than M150, which would be the cheaper option if the requirement is later reinterpreted as raw capacity.
- Increases the production cost baseline for the data tier.
- Requires post-launch measurement discipline to confirm the working set and decide whether a downgrade is justified.

### Neutral

- The service still depends on the same private networking, monitoring, and key management controls already required by
  the architecture.
- This decision does not materially change application code patterns beyond standard cache sizing and eviction policy
  tuning.

## 🔒 Compliance Considerations

- Azure Managed Redis remains deployable within the swedencentral architecture and the private-endpoint security model.
- The decision supports the project’s zero-public-endpoint expectation for data services.
- No additional compliance exception is introduced beyond those already documented for Front Door and Entra services.

## 📝 Implementation Notes

- Deploy Azure Managed Redis M200 in swedencentral with Private Endpoint access from the application VNet.
- Emit metrics and diagnostics to Azure Monitor and Log Analytics for memory pressure, latency, and failover events.
- Validate real working-set utilization after MVP launch before considering any downgrade to M150.
- Keep application cache policies conservative until production telemetry confirms sustained headroom.
