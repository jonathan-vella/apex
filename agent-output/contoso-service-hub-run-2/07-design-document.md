## 📝 1. Introduction

### 1.1 Document Purpose

This design document describes the validated Azure infrastructure design for Contoso Service Hub. Because Step 6 concluded as `validated-not-deployed`, this document captures the final approved implementation baseline rather than live deployed resource state.

| Deployment State       | Meaning                              |
| ---------------------- | ------------------------------------ |
| ✅ Validated design    | Build, lint, and review gates passed |
| ⚠️ Conditional release | Deployment prerequisites remain open |
| ❌ Live estate         | Not available in this project run    |

**Intended Audience:**

- Solution Architects
- Platform Engineering and SRE teams
- Security and Compliance stakeholders
- Delivery teams preparing production rollout

### 1.2 Project Overview

Contoso Service Hub is a greenfield, full-stack digital services platform supporting bookings, payments, digital content, and customer engagement for an EU real-estate and lifestyle ecosystem. The platform spans 15 Azure services across development, staging, and production, with GDPR as a mandatory design constraint and `swedencentral` as the primary regional anchor.

**Business Objectives:**

- Deliver an Azure-hosted digital service hub with MVP readiness for May 2026 and a 99.9% service objective.
- Support 5,000 initial users, 15,000 MAU, and growth from 50,000 annual transactions to 2,000,000 annual transactions.
- Maintain EU data residency for regional services while explicitly governing global service exceptions.

### 1.3 Design Objectives

| Objective    | Target                                   | Implementation                                                                           |
| ------------ | ---------------------------------------- | ---------------------------------------------------------------------------------------- |
| Availability | 99.9% SLA                                | Zone-redundant PostgreSQL in production, AKS self-healing, Front Door edge redundancy    |
| Performance  | p95 API latency < 500 ms                 | Front Door caching, APIM gateway, AKS autoscaling, Azure Managed Redis M200              |
| Security     | Zero public data plane for core services | Private Endpoints, delegated PostgreSQL subnet, Managed Identity, WAF, RBAC-only storage |
| Scalability  | 40x transaction growth path              | AKS node scaling, APIM scale-out path, PostgreSQL scale-up path, Redis headroom          |

### 1.4 Constraints & Assumptions

**Constraints:**

- Multi-region disaster recovery is excluded from initial scope by the RFQ.
- Azure Front Door and Entra External ID require legal approval under DR-08 before production deployment because of EU Data Boundary exceptions.

**Assumptions:**

- Production deployment will use the validated parameter set in [../../infra/bicep/contoso-service-hub-run-2/main.bicepparam](../../infra/bicep/contoso-service-hub-run-2/main.bicepparam).
- Required governance tag values, Entra tenant setup, and subscription prerequisites will be supplied before live deployment.

## 🌐 3. Networking

The network design uses a single virtual network per environment with explicit subnet separation for AKS, APIM, private data-plane access, VM workloads, and delegated PostgreSQL access. Production uses `10.0.0.0/16`, with staging and development following the `10.1.0.0/16` and `10.2.0.0/16` patterns.

Key validated networking characteristics:

- Front Door is the only internet-facing entry point.
- API Management provides the managed gateway tier for backend APIs.
- PostgreSQL uses delegated subnet integration rather than a private endpoint.
- Private Endpoints are validated for Redis, Key Vault, Blob, Files, and APIM.
- Five private DNS zones resolve data-plane traffic inside each environment.
- The design remains compliant with the tenant rule that AKS must stay under 10 agent pools; the validated cluster uses 2 pools.

| Network Element     | Validated Design                                     |
| ------------------- | ---------------------------------------------------- |
| Edge ingress        | Azure Front Door Premium + WAF                       |
| API ingress         | APIM Standard v2                                     |
| VNet                | One per environment                                  |
| NSGs                | Six logical NSG protections per environment          |
| Private DNS zones   | PostgreSQL, Redis Enterprise, Key Vault, Blob, Files |
| Data-plane exposure | Private only                                         |

## 💾 4. Storage

The storage tier separates object, file, block, and relational persistence concerns.

| Service                    | Validated Production Profile          | Purpose                                        |
| -------------------------- | ------------------------------------- | ---------------------------------------------- |
| Azure Blob Storage         | StorageV2 Hot LRS, 200 GB             | Content, assets, and application object data   |
| Azure Files                | Premium LRS, 256 GB                   | Shared SSD-backed file workloads               |
| Managed Disk               | Premium SSD P30, 256 GB               | Data disk for VM-hosted workload needs         |
| PostgreSQL Flexible Server | GP D4ds v5, 256 GB, zone-redundant HA | Core transactional store                       |
| Azure Managed Redis        | MemoryOptimized M200                  | Session, hot-path cache, and low-latency reads |

Storage-related design choices validated in Step 5:

- Blob public access is explicitly disabled.
- Shared key access is explicitly disabled on storage accounts.
- PostgreSQL requires secure transport and minimum TLS 1.2.
- Redis is configured for encrypted client protocol.
- Key Vault backs secrets and certificate handling rather than embedding credentials in code or templates.

## 💻 5. Compute

Compute is split between containerized microservices and a supporting management workload.

| Compute Element | Validated Production Profile                            | Rationale                                                               |
| --------------- | ------------------------------------------------------- | ----------------------------------------------------------------------- |
| AKS             | Standard tier, 2 user nodes on `Standard_D8s_v5`        | Primary runtime for the microservice estate                             |
| AKS system pool | `Standard_D4s_v5` class equivalent in the approved plan | Segregates system components from workload nodes                        |
| VM              | `Standard_D8s_v5` Linux                                 | Hosts workloads that do not fit the container runtime                   |
| API Management  | Standard v2                                             | API governance, gateway policy execution, and private backend mediation |

ADR alignment:

- [03-des-adr-001-container-orchestration.md](./03-des-adr-001-container-orchestration.md) confirms AKS over Container Apps for the core runtime.
- [03-des-adr-002-redis-tier.md](./03-des-adr-002-redis-tier.md) and the checked-in parameter file reconcile the cache baseline to `MemoryOptimized_M200` for production.

## 👤 6. Identity & Access

Identity is intentionally split between external and internal personas.

| Identity Domain              | Validated Design            |
| ---------------------------- | --------------------------- |
| Customer CIAM                | Microsoft Entra External ID |
| Internal workforce           | Microsoft Entra ID          |
| Service-to-service auth      | Managed Identity            |
| Infrastructure authorization | Azure RBAC                  |
| Sensitive configuration      | Azure Key Vault + RBAC      |

Important identity notes:

- Entra External ID is not provisioned by the Bicep templates and remains an external prerequisite.
- AKS, APIM, and VM workloads use Managed Identity rather than embedded secrets.
- PostgreSQL Microsoft Entra administrator configuration remains a post-deployment task.
- The tenant tag-policy drift is mitigated by setting both `technical-contact` and `tech-contact` in the validated tag object.

## 🔄 8. Backup & Disaster Recovery

The solution is designed for single-region operation with documented recovery procedures rather than active multi-region redundancy.

| Area                      | Validated Position                                  |
| ------------------------- | --------------------------------------------------- |
| Region strategy           | Single primary region: swedencentral                |
| Alternate recovery region | germanywestcentral for manual restoration scenarios |
| PostgreSQL                | PITR and zone-redundant HA in production            |
| Storage                   | Platform redundancy and service recovery features   |
| Key Vault                 | Soft delete and purge protection                    |
| DR model                  | Restore-based, not active-active                    |

The design meets the RFQ’s stated 4-hour RTO and 1-hour RPO targets on paper for the supported workload tiers, but it does not attempt to exceed that boundary with multi-region failover because that scope was explicitly excluded.

## 📊 9. Management & Monitoring

Monitoring is centralized through Log Analytics, Application Insights, Azure Monitor alerts, and an action group wired by the Bicep stack.

| Capability         | Validated Design                                          |
| ------------------ | --------------------------------------------------------- |
| Logs               | Log Analytics workspace                                   |
| App telemetry      | Workspace-based Application Insights                      |
| Alert routing      | Azure Monitor Action Group                                |
| Cost governance    | Resource-group budget with 80%, 100%, and 120% thresholds |
| Operational review | Dry-run build, lint, and E2E validation evidence          |

Operationally, the validated design is strong on observability setup and cost controls, but still expects post-deployment work to finalize dashboards, live alert tuning, and recovery rehearsals.
