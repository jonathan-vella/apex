## 📋 Executive Summary

This DR plan documents the validated recovery posture of the Contoso Service Hub design. The
platform is primarily **single-region with zone resilience**, not active-active multi-region.
Recovery therefore combines zone-redundant production services, Bicep-based rebuild procedures,
and service-native recovery features such as PostgreSQL PITR and Key Vault soft delete.

| Metric | Current | Target |
| --- | --- | --- |
| **RPO** | 1 hour target is substantially supported for PostgreSQL; partial for the wider platform | 1 hour |
| **RTO** | 4 hours target is feasible for regional rebuild with rehearsed procedures | 4 hours |
| **Availability** | 99.9% validated design goal with zone-aware production tiers | 99.9% |

### 1.2 Recovery Point Objective (RPO)

| Data Type | RPO Target | Backup Strategy |
| --- | --- | --- |
| Transactional data in PostgreSQL | 1 hour or better | Microsoft-managed backups and PITR with 35-day prod retention |
| Blob and file content | 24 hours | Storage soft delete plus application-level rehydration or copy procedures |
| Secrets and keys | Near-zero metadata loss | Key Vault soft delete and purge protection |
| Redis cache data | Best effort | Rebuild from source systems; add persistence/export automation if cache warm-up time becomes material |

## 💾 2. Backup Strategy

| Service | Validated configuration | Recovery notes |
| --- | --- | --- |
| PostgreSQL Flexible Server | 35-day backup retention in production, 14-day in non-production, production geo-redundant backup enabled | PITR creates a new server; geo-restore is available only because geo-redundant backup is enabled at creation |
| Storage Account | 14-day blob soft delete and container soft delete | Protects against accidental deletion, not full regional disaster by itself |
| Key Vault | Soft delete and purge protection for 90 days | Supports vault and secret recovery if accidental deletion occurs |
| Redis Enterprise | No persistence or export configured in code | Treat as disposable cache; recover by repopulating from PostgreSQL or upstream systems |
| AKS configuration | IaC-defined through Bicep; workloads expected from CI/CD or GitOps | Rebuild cluster from Bicep, then redeploy workloads |
| Management VM | Disks are encrypted at host, but Azure Backup is not provisioned in code | Add VM backup policy before production if the VM stores unique operational state |

The most important backup posture difference versus the original architecture narrative is the
production PostgreSQL setting. The validated Bicep module enables **geo-redundant backup**. That
improves disaster recovery but creates a governance check because backup data is copied to the
paired region.

## 🌍 3. Disaster Recovery Procedures

### 3.1 Zone Failure

1. Confirm whether the incident is limited to a single availability zone or a broader regional
   issue.
2. Allow Application Gateway, Redis, Bastion, and AKS production zone-aware services to self-heal
   before forcing manual failover actions.
3. For PostgreSQL zone-redundant HA events, monitor server state and application connectivity.
4. If a single-zone issue persists beyond the service design threshold, rebuild dependent compute
   in healthy zones using Bicep and redeploy workloads.

### 3.2 Regional Failure

1. Declare regional disaster and freeze non-essential change activity.
2. Stand up a replacement resource group in the approved recovery region using the same Bicep
   templates and environment parameters adjusted for the DR target.
3. Restore PostgreSQL from geo-redundant backup if production recovery requires database continuity.
4. Recreate Key Vault, Storage, App Gateway, APIM, AKS, Bastion, and the management VM from IaC.
5. Redeploy application workloads and rehydrate Redis from the restored data sources.
6. Update DNS and client routing only after application validation completes.

This plan assumes a rebuild-and-restore pattern. There is no active-active traffic steering or
pre-provisioned secondary regional footprint in the validated design.

## 📖 8. Recovery Runbooks

| Scenario | Runbook | Owner |
| --- | --- | --- |
| PostgreSQL logical error | Restore to a new server using PITR and redirect the application | Database engineering |
| Regional rebuild | Re-run Bicep phases in the approved recovery region and restore stateful services | Platform engineering |
| Key Vault secret deletion | Recover soft-deleted secret or vault, then revalidate workloads | Security operations |
| AKS cluster failure | Recreate cluster from Bicep and redeploy workloads from release artifacts | Service Hub engineering |

**Runbook: PostgreSQL PITR**

Trigger: Data corruption, destructive change, or accidental delete in the PostgreSQL server.

1. Confirm the last known good timestamp.
2. Restore to a new server using Azure CLI or the portal.
3. Validate schema and application smoke tests.
4. Cut application traffic to the restored server once validation completes.

Validation:

```bash
az postgres flexible-server show \
  --resource-group rg-contoso-hub-prod \
  --name psql-contoso-hub-prod-restored
```
