## 📋 Overview

This corrected Step 4 plan uses the live governance artifacts in `04-governance-constraints.md`
and `04-governance-constraints.json` as the authoritative contract for the Bicep implementation.
It supersedes the earlier placeholder assumptions that were carried into the first E2E run.

The core corrections are:

- Resource-group provisioning now assumes the live **9-tag deny baseline** and temporarily mirrors
  both `technical-contact` and `tech-contact` until the platform policy drift is resolved.
- The module contract is now **portable**: tags, budget recipients, retention, and stage selection
  are parameters or composed objects, not hardcoded project-local assumptions.
- Deployment stages are now **deterministic and cumulative**. Selecting a later stage reruns all
  prerequisite stages idempotently so required shared resources are always present.
- Storage, monitoring, and Bastion guidance now align to the live policy set and the single-region,
  same-subscription hub-spoke strategy from Step 2.
- Budgeting is redesigned around **forecast notifications at 80% / 100% / 120%** with explicit
  recipient parameters, while cost-anomaly routing is handled through parameterized platform action
  groups rather than hidden defaults.

This plan remains resource-group scoped. Tenant prerequisites such as External ID tenant creation
and Azure EU Data Boundary enablement remain out of scope for Step 4 and must be completed before
any deployment attempt.

## 🚀 Deployment Phases

> Deployment strategy: **Phased** — Foundation → Data → Edge → Platform.
>
> **Deterministic cumulative model**: later stages always include earlier stages. Example:
> `deployStage = 'edge'` deploys **foundation + data + edge** in a single idempotent run.

### Phase 1: Foundation

| Order | Module | Resources | Validation |
| ----- | ------ | --------- | ---------- |
| 1.1 | networking.bicep | VNet, subnets, NSGs, private DNS zones | Address spaces, subnet delegation, DNS links, same-subscription topology |
| 1.2 | monitoring.bicep | Log Analytics, Application Insights, explicit diagnostics helper baseline | Workspace-based telemetry, retention parameter, no duplicate diagnostics for policy-covered resources |
| 1.3 | identity.bicep | User-assigned managed identity | Principal ID and client ID available for dependent modules |
| 1.4 | budget.bicep | RG budget and forecast notifications | Forecast 80/100/120 notifications and explicit recipients validated |

**Approval Gate**: Confirm the resource-group tag payload, networking baseline, monitoring baseline, and budget recipients before any private data-plane services are added.

### Phase 2: Data

| Order | Module | Resources | Validation |
| ----- | ------ | --------- | ---------- |
| 2.1 | keyvault.bicep | Key Vault and private endpoint | Purge protection, RBAC, and private resolution |
| 2.2 | storage.bicep | Storage account, blob/file private endpoints, explicit diagnostics helper | `Standard_ZRS`, no GRS, `allowBlobPublicAccess = false`, `allowSharedKeyAccess = false` |
| 2.3 | postgresql.bicep | PostgreSQL Flexible Server and private endpoint | Entra auth, `TLSv1.2`, private-only connectivity, backup mode |
| 2.4 | redis.bicep | Redis Enterprise, database, private endpoint, explicit diagnostics helper | `TLS 1.2`, private-only access, environment sizing |

**Approval Gate**: Confirm that all data services are reachable only through private networking and that storage hardening properties are set explicitly instead of relying on post-deploy policy remediation.

### Phase 3: Edge

| Order | Module | Resources | Validation |
| ----- | ------ | --------- | ---------- |

**Approval Gate**: Confirm ingress routing, WAF rules, and APIM backend connectivity before the compute plane is attached.

### Phase 4: Platform

| Order | Module | Resources | Validation |

| 4.3 | vm.bicep | Management VM | Managed disks, Bastion-only access, identity wiring |

**Approval Gate**: Confirm workload bootstrap, management-path security, and environment isolation before handing off to Step 5 regeneration.
