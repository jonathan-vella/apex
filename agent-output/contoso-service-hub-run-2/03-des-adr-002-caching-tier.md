## 🔍 Context

The RFQ requires a 128 GB in-memory cache tier for session state, hot API responses, and
real-time application data. The Step 2 assessment concluded that Redis Enterprise E50 is the most
balanced fit for the baseline architecture because it provides 128 GB natively and avoids forcing
the initial production design into a clustered premium topology.

The cost trade study for this run is:

| Option | Capacity fit | Topology impact | Estimated monthly cost | Outcome |
| ------ | ------------ | --------------- | ---------------------- | ------- |
| Redis Enterprise E50 | Native 128 GB | Single logical tier | ~$2,520 | Selected |
| Premium P4-class design | Up to ~120 GB max | Clustering required | ~$3,800 | Rejected |

The Premium path is not only more expensive for the required requirement envelope, it also adds
sharding, operational overhead, and application-level cluster assumptions earlier than necessary.

## ✅ Decision

Use Redis Enterprise E50 as the primary cache tier.

Redis Enterprise E50 is selected because it provides the required 128 GB capacity natively,
reduces operating complexity, and costs materially less than the Premium clustered alternative for
this workload profile.

## 🔄 Alternatives Considered

- `Redis Enterprise E50`
  Pros: native 128 GB capacity, cleaner baseline topology, zone redundancy support, and room for
  Redis modules if the application later needs them.
  Cons: remains one of the largest single cost drivers in the platform.
  WAF impact: Reliability `+`, Performance `+`, Cost `-`.
- `Premium P4-class clustered design`
  Pros: uses the familiar premium service family and can be scaled through clustering.
  Cons: requires clustering to approach the stated capacity target, tops out around the same
  range, and costs more for the required footprint.
  WAF impact: Cost `-`, Operations `-`, Reliability `→`.
- `Reduce the cache requirement below 128 GB`
  Pros: lowest cost and simplest sizing.
  Cons: conflicts with the stated volumetric requirement and increases the chance of early
  eviction pressure or redesign.
  WAF impact: Cost `+`, Performance `-`, Risk `-`.

## ⚖️ Consequences

### Positive

- Meets the 128 GB requirement without redesigning the baseline around clustering.
- Simplifies initial application integration, capacity planning, and operations.
- Improves headroom for session state, hot reads, and latency-sensitive application paths.
- Preserves a cleaner path to zone-redundant production deployment in `swedencentral`.

### Negative

- Still consumes a disproportionate share of the monthly production budget.
- Requires explicit cost governance because the cache tier remains a concentrated spend area.
- Creates pressure to justify the capacity envelope with performance baselines and cache metrics.

### Neutral

- The application contract should remain portable so a later move to a smaller or clustered cache
  topology is still possible if economics change.
- The decision does not remove the need for disciplined TTL, eviction monitoring, and cache key
  design.

## 🏛️ WAF Pillar Analysis

- Security: `→` The tier decision does not materially change the security model when private
  access and TLS are enforced.
- Reliability: `↑` Native sizing reduces topology complexity and improves operational confidence.
- Performance: `↑` Larger native memory capacity reduces eviction pressure on hot paths.
- Cost: `↓` E50 is still a major cost driver even though it is cheaper than the Premium option.
- Operations: `↑` Avoiding mandatory clustering keeps the first implementation and support model
  simpler.

## 🔒 Compliance Considerations

- Deploy in `swedencentral` and keep access private through Private Link.
- Apply the same GDPR and PCI-DSS network isolation patterns used for PostgreSQL and storage.
- Treat cached session and PII-adjacent data as regulated data for TTL, purge, and logging rules.

## 📝 Implementation Notes

- Use private endpoints and disable unnecessary public exposure.
- Instrument memory pressure, hit ratio, evictions, connection count, and command latency from
  day one.
- Keep cache keys and data models portable so the platform can move to a different topology later
  if cost pressure changes the trade-off.
- Do not treat Redis as a system of record.
