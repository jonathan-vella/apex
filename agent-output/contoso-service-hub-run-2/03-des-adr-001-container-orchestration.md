## 🔍 Context

GAP-03 in the architecture assessment required a final container orchestration decision for a platform built around
containerized microservices, private networking, and sustained business traffic rather than sporadic burst traffic.
The target platform fronts all APIs through Azure Front Door Premium and API Management Standard v2, runs private
data services behind Private Endpoints, and must support three environments with repeatable deployment pipelines.

Azure Container Apps was a credible option because it lowers platform operations overhead and can be a good fit for
smaller service estates. That trade-off weakens once the design needs Kubernetes-native network controls, stronger
workload isolation, predictable steady-state cost at sustained load, and a clearer path for multi-service release
coordination across Development, Staging, and Production.

## ✅ Decision

Use Azure Kubernetes Service Standard as the primary container orchestration platform for Contoso Service Hub.

Production will run on AKS Standard in swedencentral with zone-redundant worker capacity, managed identity-based
access to downstream services, and private connectivity to PostgreSQL Flexible Server, Azure Managed Redis, Blob
Storage, Azure Files, and Key Vault. Azure Container Apps remains a future optimization candidate for isolated
auxiliary workloads, but not for the core service hub runtime.

## 🔄 Alternatives Considered

| Option                           | Pros                                                                                                                | Cons                                                                                                                                 | WAF Impact                                                 |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------- |
| **AKS Standard**                 | Full Kubernetes control, network policies, strong workload isolation, predictable scaling for a microservice estate | Higher day-2 operational overhead and a stronger platform engineering requirement                                                    | Security: +, Reliability: +, Performance: +, Operations: - |
| Azure Container Apps             | Lower operational burden, simpler deployment model, good developer experience for smaller service sets              | Less control over cluster-level policy, weaker fit for fine-grained runtime isolation, less predictable cost at sustained throughput | Operations: +, Cost: +, Security: -                        |
| Mixed App Service and VM hosting | Lower platform complexity for a smaller application footprint                                                       | Does not fit the RFP container-engine direction and breaks the microservice orchestration model chosen in Step 2                     | Cost: +, Performance: -, Operations: +                     |

## ⚖️ Consequences

### Positive

- Aligns the runtime with the Step 2 architecture pattern of N-tier microservices on AKS with managed PaaS data
  services.
- Provides the strongest fit for private networking, workload identity, network policies, and controlled east-west
  traffic flows.
- Supports progressive deployment and environment promotion through GitHub Actions without changing the runtime model
  between Development, Staging, and Production.
- Gives the team a clean path to introduce KEDA, admission controls, and stricter namespace isolation as the platform
  grows.

### Negative

- Increases platform operations effort compared with Container Apps.
- Requires stronger Kubernetes expertise for node pool upgrades, security policy enforcement, and operational support.
- Carries a higher baseline compute and support footprint than a simpler serverless container runtime.

### Neutral

- API Management, Front Door WAF, and the private data plane remain mandatory regardless of the orchestration choice.
- Choosing AKS does not change the single-region scope; reliability remains bounded by the RFQ decision to defer
  multi-region disaster recovery.

## 🏛️ WAF Pillar Analysis

| Pillar      | Impact | Notes                                                                                                                         |
| ----------- | ------ | ----------------------------------------------------------------------------------------------------------------------------- |
| Security    | ↑      | Private cluster controls, network policy support, and workload identity are a better fit for the required zero-trust posture. |
| Reliability | ↑      | Zone-redundant node pools and Kubernetes self-healing improve service resilience inside the single-region boundary.           |
| Performance | ↑      | HPA, cluster autoscaler, and workload placement controls give a stronger scaling model for sustained platform load.           |
| Cost        | ↓      | AKS introduces more fixed platform cost than Container Apps for the same service baseline.                                    |
| Operations  | ↓      | Day-2 operations are materially more complex and demand stronger platform ownership.                                          |

## 🔒 Compliance Considerations

- AKS supports the GDPR-driven private network segmentation required by the architecture assessment.
- Managed identity and private endpoint access preserve the design goal of avoiding secret sprawl and public data-plane
  exposure.
- This decision does not remove the architecture’s compliance blocker around Front Door and Entra External ID legal
  approval under DR-08.

## 📝 Implementation Notes

- Provision AKS Standard with zone-aware worker capacity in swedencentral and keep lower environments topology-aligned.
- Enforce Kubernetes network policies, Azure Policy for AKS, and workload identity before onboarding application
  namespaces.
- Route all data-service traffic over Private Endpoints and keep public network access disabled on the data plane.
- Use GitHub Actions to promote immutable container releases from Development to Staging to Production.
