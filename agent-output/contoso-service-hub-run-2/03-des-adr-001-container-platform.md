## 🔍 Context

The RFQ requires a managed container platform and explicitly mandates managed Kubernetes in
RFQ §4.1 #10. The Step 2 architecture assessment therefore narrowed the compute platform choice
to AKS versus Azure Container Apps, with App Service retained as a simpler alternative for select
web workloads rather than the platform baseline.

The platform must support fifteen Azure services, API-first microservices, private connectivity to
PostgreSQL, Redis, Storage, and Key Vault, and a production operating model that can scale from
MVP launch to materially higher transaction volumes without a platform rewrite.

## ✅ Decision

Use AKS Standard as the primary container platform for Contoso Service Hub.

AKS is selected because it satisfies the managed Kubernetes mandate, supports the expected
microservices growth profile, and provides the networking and policy controls required for a
GDPR-focused, enterprise Azure platform.

## 🔄 Alternatives Considered

- `AKS Standard`
  Pros: satisfies the RFQ mandate, supports advanced networking and policy control, and fits the
  expected microservices operating model.
  Cons: higher operational overhead and a higher steady-state platform cost than lighter PaaS
  options.
  WAF impact: Security `+`, Reliability `+`, Performance `+`, Cost `-`, Operations `-`.
- `Azure Container Apps`
  Pros: lower operational burden, simpler autoscaling, and faster MVP delivery for a smaller
  estate.
  Cons: does not satisfy the explicit managed Kubernetes requirement and offers less platform
  control for a large multi-service estate.
  WAF impact: Operations `+`, Cost `+`, Security `→`, Reliability `→`.
- `App Service`
  Pros: simple web hosting model, mature deployment surface, and lower day-2 platform overhead.
  Cons: diverges from the requested container platform pattern and weakens portability for the
  planned service decomposition.
  WAF impact: Cost `+`, Operations `+`, Architecture fit `-`.

## ⚖️ Consequences

### Positive

- Aligns directly with RFQ §4.1 #10 and removes ambiguity around the primary container platform.
- Provides stronger control over ingress, workload isolation, pod scaling, and network policy.
- Supports future platform growth without changing the core runtime model.
- Fits the private endpoint and managed identity baseline described in Step 2.

### Negative

- Increases platform operations complexity relative to Azure Container Apps or App Service.
- Introduces a higher always-on cluster cost floor.
- Requires stronger engineering discipline around upgrades, node pools, security baselines,
  observability, and GitOps.

### Neutral

- Azure Functions or App Service can still be introduced later for narrowly scoped workloads if a
  later implementation constraint justifies that split.
- The decision does not remove the need for capacity testing and autoscaling validation before
  production cutover.

## 🏛️ WAF Pillar Analysis

- Security: `↑` AKS supports stronger segmentation, policy enforcement, and private connectivity.
- Reliability: `↑` Multi-node orchestration, rolling updates, and autoscaling improve resilience.
- Performance: `↑` AKS gives predictable resource control for CPU, memory, and ingress tuning.
- Cost: `↓` AKS has a higher baseline cost than Container Apps for comparable early-stage load.
- Operations: `↓` The team inherits more platform operations work around cluster governance.

## 🔒 Compliance Considerations

- Supports GDPR-aligned regional deployment in `swedencentral` with private connectivity to the
  regulated data tier.
- Enables PCI-DSS segmentation by separating ingress, application, and data access paths.
- Supports managed identity and Key Vault integration so secrets do not need to live in code or
  CI/CD variables.

## 📝 Implementation Notes

- Use separate system and user node pools in production.
- Route ingress through Application Gateway and API Management rather than exposing services
  directly.
- Enable Azure Policy for AKS, Defender for Containers, cluster autoscaler, health probes, and
  image governance as implementation baselines.
- Preserve the option to attach Azure Functions later for event-driven tasks that do not justify
  long-running pods.
