# ADR 0001 — Azure Service Bus + Event Grid for Async Order Processing

## Status

Accepted

## Context

The order-processing system requires:

1. **Reliable command fan-out** — `orders-api` must reliably deliver `orders`
   topic messages to `orders-worker` with at-least-once delivery semantics and
   dead-letter support.
2. **Change-data-capture propagation** — when Cosmos DB records change (order
   state transitions), `notifications-worker` must be triggered without polling.

Two integration services are therefore needed:

- **Azure Service Bus (Standard)** — provides the `orders` topic with two
  subscriptions; chosen over Azure Storage Queues because topics enable future
  fan-out to additional subscribers without schema changes.
- **Event Grid system topic** — provided natively by Cosmos DB via the
  "Azure Cosmos DB" event source; delivers change-feed events to
  `notifications-worker` via a push model, eliminating the need for a
  change-feed processor running inside `orders-worker`.

## Decision

Use Azure Service Bus (Standard tier, `orders` topic, 2 subscriptions) for
command messaging and an Event Grid system topic bound to the Cosmos DB
account for change-data-capture notifications.

`orders-api` publishes to Service Bus over AMQP; `orders-worker` subscribes
via a Service Bus–triggered Container App (KEDA `azure-servicebus` scaler).
The Event Grid system topic routes Cosmos DB events to `notifications-worker`
via a Container App HTTP endpoint (KEDA `azure-eventhubs`-compatible trigger or
direct HTTP).

## Consequences

### Reliability (WAF pillar)

- Service Bus Standard provides 256 KB max message size, 80 GB topic storage,
  and dead-letter queues — sufficient for order commands.
- Event Grid guarantees at-least-once delivery with configurable retry policy.
- Both services are zone-redundant within the Sweden Central region.

### Performance Efficiency (WAF pillar)

- AMQP over port 5671 (TLS) for Service Bus avoids HTTP polling overhead.
- Event Grid push eliminates the 1-5 s polling latency of a change-feed
  processor loop.

### Cost Optimisation (WAF pillar)

- Service Bus Standard is priced per operation; the Consumption-tier Container
  Apps Environment scales `orders-worker` and `notifications-worker` to zero
  when idle, avoiding idle compute charges.
- Event Grid system topics are free (no charge for the system topic resource
  itself); only event deliveries are billed.

### Operational Excellence (WAF pillar)

- Dead-letter queues on the Service Bus `orders` topic provide a durable
  audit trail and enable manual replay of failed order commands.
- Application Insights traces both the Service Bus trigger and Event Grid
  delivery, giving end-to-end distributed tracing across the async boundary.

### Trade-offs accepted

- Service Bus Standard does not support geo-disaster recovery (requires
  Premium). Accepted for this workload; upgrade path to Premium is documented.
- Event Grid system topic does not support batching to `notifications-worker`;
  each Cosmos DB change emits an individual event. Acceptable at current scale.
