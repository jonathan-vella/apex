## 🔍 Context

The architecture assessment finalized the core runtime around Azure Front Door Premium, API Management in
internal VNet mode, AKS-hosted microservices, and a private data plane for PostgreSQL Flexible Server,
Azure Cache for Redis Premium, Storage, and Key Vault. The platform must support 15 microservices,
three environments, managed identity-based access, zone-aware scaling, and stricter network controls than
an MVP-only serverless container platform would typically require.

Azure Container Apps remained a serious alternative because it reduces operational overhead and simplifies
deployment. That trade-off becomes less attractive once the design requires D8s_v5 node sizing, Kubernetes
network policy support, predictable multi-service placement, namespace-level isolation, and a scaling path
for growth from 50,000 to 2,000,000 annual transactions without re-platforming.

## ✅ Decision

Use Azure Kubernetes Service Standard as the primary container platform for Contoso Service Hub.

Production will run on AKS Standard in swedencentral with three Standard_D8s_v5 worker nodes distributed
across availability zones, managed identity-based access to downstream services, and API ingress mediated
through Azure API Management in internal VNet mode. Azure Container Apps is not selected for the core
runtime, but remains a potential future fit for isolated auxiliary workloads that do not require full
Kubernetes controls.

## 🔄 Alternatives Considered

| Option                    | Pros                                                                                     | Cons                                                                                                         | WAF Impact                                                 |
| ------------------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------- |
| **AKS Standard**          | Full Kubernetes control, strong networking model, predictable node sizing, mature ops    | Higher day-2 platform complexity, requires Kubernetes engineering capability                                 | Security: +, Reliability: +, Performance: +, Operations: - |
| Azure Container Apps      | Lower operational burden, simpler deployment model, good fit for smaller service estates | Less control over networking and runtime isolation, weaker fit for explicit node sizing and K8s policy needs | Operations: +, Cost: +, Security: -                        |
| App Service or mixed PaaS | Lowest platform complexity for a simpler application footprint                           | Does not meet the managed Kubernetes direction and orchestration needs of the architecture                   | Cost: +, Performance: -, Operations: +                     |

## ⚖️ Consequences

### Positive

- Aligns the implementation with the architecture decision already captured in the Step 2 assessment.
- Supports private networking, workload identity, Kubernetes-native autoscaling, and namespace isolation.
- Preserves a clear path for progressive delivery, GitHub Actions-driven promotions, and policy enforcement.
- Keeps the platform ready for later additions such as KEDA, admission controls, and pod disruption budgets.

### Negative

- Increases operational complexity relative to Azure Container Apps.
- Requires AKS lifecycle ownership for node image updates, upgrades, and security policy enforcement.
- Carries a higher fixed compute and support baseline than a simpler managed container runtime.

### Neutral

- Front Door, APIM, private endpoints, and the observability stack remain mandatory regardless of runtime choice.
- This decision does not change the single-region scope or the deferred multi-region disaster recovery boundary.

## 🏛️ WAF Pillar Analysis

| Pillar      | Impact | Notes                                                                                                             |
| ----------- | ------ | ----------------------------------------------------------------------------------------------------------------- |
| Security    | ↑      | AKS better supports private-cluster controls, network policies, and workload identity for the zero-trust posture. |
| Reliability | ↑      | Zone-aware node pools and Kubernetes self-healing strengthen resilience within the single-region design.          |
| Performance | ↑      | HPA and cluster autoscaler provide the best fit for the platform's sustained and burst traffic profile.           |
| Cost        | ↓      | AKS has a higher baseline cost than Container Apps for the same service estate.                                   |
| Operations  | ↓      | Day-2 ownership is materially heavier and requires platform engineering maturity.                                 |

## 🔒 Compliance Considerations

- Supports the GDPR-driven private network segmentation already required by the architecture.
- Managed identity and private endpoint access reduce secret sprawl and public data-plane exposure.
- Keeps payment-adjacent workloads behind the API gateway and private backend boundaries assumed for SAQ-A scope.

## 📝 Implementation Notes

- Provision AKS Standard with three Standard_D8s_v5 worker nodes across availability zones in swedencentral.
- Enforce workload identity, Kubernetes network policies, and Azure Policy for AKS before onboarding services.
- Route all PostgreSQL, Redis, Storage, and Key Vault access through private endpoints only.
- Configure HPA and cluster autoscaler in Step 4 planning, with a production scaling range of 3 to 6 nodes.
