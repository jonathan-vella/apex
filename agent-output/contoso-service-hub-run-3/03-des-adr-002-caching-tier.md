## 🔍 Context

The requirements originally implied a 128 GB cache tier, which would have pushed the design toward a much more
expensive Enterprise-class Redis deployment. The architecture assessment revalidated the real MVP demand profile:
5,000 initial users, moderate session-state pressure, and a hot-path caching layer for availability, content, and
personalization reads rather than a sustained 128 GB working set on day one.

The resulting analysis showed that Azure Cache for Redis Premium P4 provides 53 GB of cache memory with enough
headroom for the MVP, materially lower monthly cost, and an acceptable operational model for a private, monitored
data service. The design still needs a defined growth trigger because the business expects strong expansion and a
40x transaction increase within the planning horizon.

## ✅ Decision

Use Azure Cache for Redis Premium P4 as the default MVP cache tier for Contoso Service Hub.

Premium P4 is selected as the initial production baseline because it satisfies the MVP workload with a strong
cost-performance balance. The architecture will retain an upgrade path to Enterprise-class Redis once sustained
memory utilization exceeds 70 percent or transaction growth materially changes the working-set profile.

## 🔄 Alternatives Considered

| Option                    | Pros                                                                                        | Cons                                                                                                   | WAF Impact                             |
| ------------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ | -------------------------------------- |
| **Redis Premium P4**      | Strong MVP fit, materially lower cost, private networking support, simpler than over-sizing | 53 GB is below the original nominal requirement and requires monitoring discipline for growth          | Cost: +, Performance: +, Operations: + |
| Redis Premium P5          | Larger memory footprint and more headroom than P4                                           | More expensive than P4 and still harder to justify at MVP scale                                        | Performance: +, Cost: -                |
| Redis Enterprise          | Highest ceiling and strategic long-term feature set                                         | Significant monthly premium before the workload actually requires it                                   | Reliability: +, Cost: -, Operations: → |
| Self-managed Redis on AKS | Maximum control and packaging flexibility                                                   | Highest operational burden, weakest fit for the managed-service posture, avoidable platform complexity | Cost: →, Reliability: -, Operations: - |

## ⚖️ Consequences

### Positive

- Reduces monthly cache spend versus Enterprise sizing while remaining aligned to the MVP workload profile.
- Preserves a managed PaaS service with private endpoint access, monitoring, and Azure-native operations.
- Avoids over-provisioning the data tier before actual workload telemetry is available.
- Creates a clean, criteria-based upgrade path instead of forcing a premature top-tier selection.

### Negative

- Introduces the need to actively monitor memory pressure and hit ratio as adoption grows.
- May require a future re-tier to Enterprise once sustained utilization passes the agreed threshold.
- Provides less raw headroom than a deliberately over-sized Enterprise deployment.

### Neutral

- The application still needs careful cache key design, TTL management, and eviction policy tuning.
- Private endpoint, diagnostics, and backup-related controls remain required regardless of tier.

## 🏛️ WAF Pillar Analysis

| Reliability | →      | Reliable for the MVP baseline, but long-term resilience depends on timely re-tiering before sustained saturation.   |
| Performance | ↑      | Delivers low-latency cache access for the current platform profile without unnecessary oversizing.                  |
| Cost        | ↑      | This is the strongest cost-optimization decision in the design, saving substantial monthly spend versus Enterprise. |

## 🔒 Compliance Considerations

- Keeps cache data inside swedencentral and inside the EU-only residency boundary required for GDPR.
- Preserves the private endpoint pattern already mandated for all data services.
- Does not introduce a new compliance exception or public data path.

## 📝 Implementation Notes

- Deploy Redis Premium P4 in swedencentral with private endpoint connectivity from the application VNet.
- Enable diagnostics to Azure Monitor and Log Analytics for memory utilization, hit ratio, latency, and failover.
- Define an upgrade trigger to Enterprise when sustained memory utilization exceeds 70 percent over a representative
  peak period.
- Reassess tiering during Step 4 planning and again after MVP launch once real cache telemetry is available.
