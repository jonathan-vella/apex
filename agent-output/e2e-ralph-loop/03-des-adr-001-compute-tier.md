## 🔍 Context

Nordic Fresh Foods Lite is a simple greenfield Azure web ordering platform with a single production
environment in `swedencentral`, GDPR-driven EU data residency, and a hard infrastructure budget of
less than €500 per month. The architecture assessment recommends Azure App Service for the web tier,
with an expected workload of about 500 users, fewer than 50 concurrent users, and roughly 50
orders per day.

The key compute decision is whether to optimize the first production release for the lowest
practical paid tier or to buy more scale and platform features before they are needed.

## ✅ Decision

Use **Azure App Service B1 Linux** as the initial production compute tier for the application.

This tier is selected because it is the lowest practical App Service SKU that aligns with the
assessment constraints:

- supports a production web workload without the limits of the free/shared tiers
- fits easily inside the project cost envelope, with the assessment estimating about €11.17/month
- provides a straightforward upgrade path to `S1` if autoscale, deployment slots, or more sustained
  capacity are required later

## 🔄 Alternatives Considered

| Option                             | Pros                                                                    | Cons                                                                                                         | WAF Impact                           |
| ---------------------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------ |
| Free or Shared App Service tier    | Lowest possible upfront cost                                            | Not appropriate for steady production usage, tighter platform limits, weaker operational posture             | Cost +, Reliability -, Operations -  |
| Standard S1 App Service            | Autoscale support, deployment slots, more operational flexibility       | Materially higher monthly cost for a workload that does not need those features yet                          | Reliability +, Operations +, Cost -- |
| Azure Container Apps               | Better burst scaling model and modern container path                    | More platform complexity than required for this small MVP and no benefit to the current workload assumptions | Performance +, Operations -, Cost -  |
| **Selected: App Service B1 Linux** | Lowest practical production tier, simple operations, clear upgrade path | Single instance and no autoscale headroom at this tier                                                       | Cost ++, Operations +, Reliability - |

## ⚖️ Consequences

### Positive

- Keeps the compute layer aligned with the project goal of staying far below the €500 monthly cap
- Minimizes operational complexity for a simple single-environment MVP
- Preserves a clean scale-up path to higher App Service tiers without changing the application model

### Negative

- Limits immediate headroom for traffic spikes because the tier is intentionally conservative
- Defers capabilities such as autoscale and deployment slots to a later upgrade
- Increases the need for monitoring thresholds so the team knows when to move to `S1`

### Neutral

- The decision optimizes for current demand rather than the 12-month growth projection
- Future upgrades remain operationally simple because the selected platform stays within App Service

## 🔒 Compliance Considerations

- GDPR: The compute tier does not change the requirement that application and data services remain in
  EU regions, specifically `swedencentral` for this workload.
- Security baseline: The selected tier must still enforce HTTPS-only access, TLS 1.2 minimum,
  Managed Identity, and Key Vault-based secret retrieval.
- Auditability: Application telemetry must flow into Application Insights and Log Analytics so
  availability and performance signals are retained for incident review.

## 📝 Implementation Notes

- Deploy the app on Linux App Service B1 in the single production environment defined by the
  assessment.
- Configure alerts for CPU, memory pressure, response time p95, and HTTP 5xx rate so the upgrade
  trigger is evidence-based.
- Revisit this decision if either of these conditions is reached:
  - sustained concurrency rises above the current `<50` target envelope
  - response time or availability targets are repeatedly missed under normal production load
