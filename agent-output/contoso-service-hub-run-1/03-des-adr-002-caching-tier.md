## 🔍 Context

The platform requires a 128 GB in-memory cache for sessions, booking availability, and high-frequency read acceleration. The older Azure Cache for Redis Enterprise E50 matched the capacity target, but that tier retires in March 2027, which falls inside the project’s three-year contract window. A tier decision that depends on a retiring SKU would introduce migration risk during the contracted service term.

Premium-tier Redis options were reviewed, but P4 and P5 remain under the stated requirement unless the design accepts clustering or an undersized deployment. That creates either capacity risk or extra operational complexity for a platform that already has a demanding container and network topology.

## ✅ Decision

Use Azure Managed Redis M100 with 128 GB capacity as the production cache tier.

This choice meets the exact capacity target, avoids the retirement risk associated with Enterprise E50, and preserves a managed service model that fits the project’s private networking, monitoring, and compliance posture.

## ⚖️ Consequences

### Positive

- Meets the 128 GB requirement without shrinking scope or relying on a retiring SKU.
- Avoids a forced cache-platform migration inside the three-year contract period.
- Preserves a managed service operating model with private access, diagnostics, and enterprise supportability.
- Simplifies application design relative to clustered or sharded Premium-tier workarounds.

### Negative

- Costs more than the Premium-tier alternatives that were considered.
- Requires the team to align application connectivity, monitoring, and failover assumptions to the Azure Managed Redis service model.
- Raises the production cost baseline for the platform’s data tier.

### Neutral

- The service still requires strict network and identity controls; moving to M100 does not reduce the need for private endpoints and managed authentication.
- Cache sizing should still be reviewed periodically against real production utilization after launch.

## 🔒 Compliance Considerations

- Supports GDPR residency requirements by staying in the EU-region architecture with private access patterns.
- Aligns with the project’s zero-public-endpoint data-plane stance when deployed behind a private endpoint.
- Reduces contract-term risk that could otherwise force a compliance-sensitive migration during service operation.

## 📝 Implementation Notes

- Deploy Azure Managed Redis M100 in swedencentral with private endpoint connectivity from the application network.
- Disable legacy local-auth patterns where supported and prefer managed identity or Entra-backed access controls.
- Emit metrics and diagnostics to Azure Monitor and Log Analytics for hit ratio, latency, failover events, and memory pressure.
- Revalidate real cache utilization after launch, but keep the initial platform design anchored to the full 128 GB requirement.
