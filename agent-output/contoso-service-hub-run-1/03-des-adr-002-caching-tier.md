## 🔍 Context

The RFQ requires a 128 GB in-memory cache tier. The Step 2 assessment confirmed that Premium P4
does not provide 128 GB natively and would require clustering to approximate the target capacity.
That alternative lowers cost materially, but adds configuration and operational complexity.

The architecture also relies on the cache for session data, hot API responses, and application
latency control across AKS-hosted services.

## ✅ Decision

Use Azure Managed Redis Enterprise E100 as the primary cache tier.

Enterprise E100 is selected because it provides 128 GB natively, avoids a forced cluster topology
for the baseline requirement, and better matches the design preference for a simpler production
operating model even though it is the largest single cost driver in the solution.

## 🔄 Alternatives Considered

- `Enterprise E100`
  Pros: native 128 GB capacity, cleaner baseline topology, and zone redundancy support.
  Cons: highest monthly cost in the platform and a major budget concentration risk.
  WAF impact: Reliability `+`, Performance `+`, Cost `-`.
- `Premium P4 with clustering`
  Pros: significantly lower monthly cost and familiar Redis operations.
  Cons: does not satisfy 128 GB natively, adds sharding complexity, and weakens the
  preferred operating model.
  WAF impact: Cost `+`, Operations `-`, Reliability `→`.
- `Reduce cache requirement below 128 GB`
  Pros: lowest cost and simplest sizing.
  Cons: conflicts with the stated volumetric requirement and increases the chance of
  cache churn or redesign.
  WAF impact: Cost `+`, Performance `-`, Risk `-`.

## ⚖️ Consequences

### Positive

- Meets the 128 GB requirement without redesigning the cache layer around clustering.
- Simplifies initial application integration and capacity planning for the production baseline.
- Improves headroom for hot-data use cases and read-heavy API performance.
- Preserves the option for zone-redundant deployment in the target region.

### Negative

- Consumes a disproportionate share of the monthly production budget.
- Reduces flexibility if cost pressure later forces a redesign around smaller or sharded cache
  tiers.
- Requires explicit cost governance because the cache tier dominates run-rate spend.

### Neutral

- The logical application contract should still be designed so a later migration to a clustered
  topology remains possible if economics change.
- The choice does not remove the need for cache TTL discipline, eviction monitoring, and capacity
  dashboards.

## 🏛️ WAF Pillar Analysis

- Security: `→` The tier decision does not materially change the security model when
  private access and TLS are enforced.
- Reliability: `↑` Native sizing reduces topology complexity and improves operational
  confidence for failover and scaling.
- Performance: `↑` Larger native memory capacity reduces eviction pressure and supports
  low-latency access for hot paths.
- Cost: `↓` E100 is materially more expensive than Premium P4 clustered alternatives.
- Operations: `↑` Avoiding a mandatory clustered baseline keeps the first implementation
  and support model simpler.

## 🔒 Compliance Considerations

- Deploy in `swedencentral` and keep connectivity private through Private Link.
- Apply the same GDPR and PCI-DSS network isolation patterns as the rest of the data tier.
- Treat cached PII and session material as regulated data for retention, TTL, and purge handling.

## 📝 Implementation Notes

- Use private endpoints and disable unnecessary public exposure.
- Instrument memory pressure, hit ratio, evictions, connection count, and command latency from day
  one.
- Keep cache keys and data models portable so the platform can move to a clustered Premium design
  later if Contoso chooses to optimize cost.
- Set TTL defaults deliberately and avoid using Redis as a long-term system of record.
