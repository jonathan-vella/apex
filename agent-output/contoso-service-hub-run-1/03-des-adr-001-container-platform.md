## 🔍 Context

The Contoso Service Hub RFP explicitly calls for a container engine aligned to managed Kubernetes, while the target platform spans 15 core cloud services plus supporting DevOps and observability integrations. The architecture requires a private cluster, fine-grained network segmentation, workload isolation, private endpoint connectivity to PostgreSQL, Redis, Storage, and Key Vault, and a scaling path that can absorb growth from 50,000 to 2,000,000 annual transactions.

Azure Container Apps was considered because it lowers platform operations overhead, but the combination of private-cluster requirements, three availability zones, GitHub Enterprise-driven release automation, and a multi-service platform with stricter networking and policy controls pushes the design toward a full Kubernetes control plane.

## ✅ Decision

Use Azure Kubernetes Service as the primary container platform for Contoso Service Hub.

The production design uses a private AKS cluster in swedencentral with three user nodes distributed across three availability zones, separate ingress and management paths, managed identity-based access to downstream services, and Kubernetes-native deployment controls for the platform’s service estate.

## 🔄 Alternatives Considered

| Option                                | Pros                                                                                                                  | Cons                                                                                                                          | WAF Impact                                 |
| ------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| Azure Kubernetes Service              | Full Kubernetes control, mature networking, private cluster support, predictable scaling for a multi-service platform | Higher operational burden, upgrade planning, requires Kubernetes expertise                                                    | Security: +, Reliability: +, Operations: - |
| Azure Container Apps                  | Lower ops overhead, simpler deployment model, good fit for smaller microservice sets                                  | Less control over network topology, weaker fit for private-cluster expectations, less precise runtime isolation               | Cost: +, Operations: +, Security: -        |
| App Service or mixed PaaS web hosting | Lowest platform complexity for a smaller application footprint                                                        | Does not fit the RFP container-engine direction and does not provide the workload orchestration model needed for the platform | Cost: +, Performance: -, Operations: +     |

## ⚖️ Consequences

### Positive

- Satisfies the RFP language around managed Kubernetes and container orchestration.
- Supports private-cluster deployment, network policies, namespace isolation, and tighter east-west controls.
- Gives the team full control over deployment patterns, autoscaling behavior, workload scheduling, and service composition.
- Aligns well with GitHub Enterprise-based CI/CD, GitOps, and progressive delivery practices.

### Negative

- Increases platform operations complexity relative to Container Apps.
- Requires Kubernetes platform engineering skills for upgrades, node pool management, policy enforcement, and incident response.
- Raises the baseline compute and support cost versus a simpler managed container runtime.

### Neutral

- Platform guardrails such as Entra-integrated RBAC, workload identity, and policy enforcement become mandatory parts of the implementation, not optional enhancements.
- The choice does not remove the need for strong API gateway, edge security, or observability controls; it makes those controls more important.

## 🏛️ WAF Pillar Analysis

| Pillar      | Impact | Notes                                                                                                                        |
| ----------- | ------ | ---------------------------------------------------------------------------------------------------------------------------- |
| Security    | ↑      | Private cluster support, network policy, workload identity, and namespace isolation fit the zero-trust target state.         |
| Reliability | ↑      | Multi-zone node pools and Kubernetes self-healing improve resilience when paired with disciplined upgrade strategy.          |
| Performance | ↑      | HPA, cluster autoscaler, and workload placement controls support the projected growth profile better than a simpler runtime. |
| Cost        | ↓      | AKS carries a higher fixed operational and compute baseline than Container Apps for this platform shape.                     |
| Operations  | ↓      | Day-2 operations are materially more complex and require a stronger platform team.                                           |

## 🔒 Compliance Considerations

- Supports GDPR-oriented network segmentation by keeping the container platform private and colocated with EU-resident data services.
- Better fits the payment-adjacent security posture because policy, RBAC, admission controls, and network isolation can be enforced at cluster level.
- Keeps the design compatible with the project’s SAQ-A tokenized payment scope by limiting direct exposure of backend services.

## 📝 Implementation Notes

- Provision AKS as a private cluster with Entra-integrated RBAC and local accounts disabled.
- Use separate system and user node pools, with three user nodes across three availability zones in production.
- Enforce Azure Policy for AKS, Kubernetes network policies, and restricted pod security standards before onboarding workloads.
- Integrate GitHub Enterprise pipelines through managed identity and a GitOps-capable deployment model.
- Route all data-service access through private endpoints and keep public network access disabled on the data plane.
