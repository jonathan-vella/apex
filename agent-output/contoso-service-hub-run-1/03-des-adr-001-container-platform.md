## 🔍 Context

The RFQ requires a managed container platform and explicitly calls out managed Kubernetes in scope.
The Step 2 assessment therefore narrowed the compute decision to Azure Kubernetes Service (AKS)
versus Azure Container Apps, with App Service plus Functions retained only as a secondary
alternative for non-containerised workloads.

The platform must support an API-first workload with growth from 500 to 2,000 concurrent users,
private connectivity to PostgreSQL, Redis, Storage, and Key Vault, and enterprise controls for
network segmentation, workload isolation, and CI/CD.

## ✅ Decision

Use AKS as the primary container platform for Contoso Service Hub.

AKS is selected because it satisfies the RFQ mandate for managed Kubernetes, supports the expected
microservices growth profile, and aligns with the architecture assessment decision log.
Azure Container Apps remains a valid alternative for simpler services, but not as the primary
platform for this project.

## 🔄 Alternatives Considered

- `AKS`
  Pros: meets the managed Kubernetes mandate, supports advanced networking, and fits
  enterprise platform controls.
  Cons: higher operational overhead and a higher steady-state platform cost than simpler
  PaaS options.
  WAF impact: Security `+`, Reliability `+`, Performance `+`, Cost `-`, Operations `-`.
- `Azure Container Apps`
  Pros: lower operational burden, simpler autoscaling, and faster MVP delivery.
  Cons: does not satisfy the explicit managed Kubernetes requirement and offers less
  platform-level control.
  WAF impact: Operations `+`, Cost `+`, Security `→`, Reliability `→`.
- `App Service plus Azure Functions`
  Pros: strong PaaS simplicity and a good fit for selected web or event workloads.
  Cons: diverges from the requested container platform pattern and weakens workload
  portability.
  WAF impact: Cost `+`, Operations `+`, Performance `→`, Architecture fit `-`.

## ⚖️ Consequences

### Positive

- Aligns directly with the RFQ requirement for managed Kubernetes.
- Provides stronger control over ingress, workload isolation, pod scaling, and network policy.
- Supports future decomposition into independently deployable services without changing the core
  platform.
- Fits private endpoint connectivity and zero-secret patterns through managed identity.

### Negative

- Increases operational complexity relative to Azure Container Apps.
- Introduces a higher always-on cost floor for the cluster and supporting operational tooling.
- Requires stronger platform engineering maturity around node pools, upgrades, observability,
  and security baselines.

### Neutral

- The project can still use Azure Functions later for narrowly scoped serverless tasks without
  changing the primary container platform decision.
- Some edge-facing or admin workloads may still land on other Azure compute services if a later
  implementation constraint justifies that split.

## 🏛️ WAF Pillar Analysis

- Security: `↑` AKS supports stronger segmentation, policy enforcement, and private
  connectivity than lighter PaaS options.
- Reliability: `↑` Multi-node orchestration, rolling updates, and autoscaling improve
  resilience for the core tier.
- Performance: `↑` AKS gives predictable resource control for CPU, memory, and ingress
  tuning under the projected growth profile.
- Cost: `↓` AKS has a higher baseline cost than Container Apps for comparable early-stage
  throughput.
- Operations: `↓` The team inherits more platform operations work around upgrades,
  node pools, and cluster governance.

## 🔒 Compliance Considerations

- Supports GDPR-aligned regional deployment in `swedencentral` with private endpoints to data
  services.
- Enables PCI-DSS-oriented segmentation by separating ingress, application, and data-access paths.
- Supports managed identity and Key Vault integration to avoid application secrets in code or
  pipelines.

## 📝 Implementation Notes

- Use at least separate system and user node pools in production.
- Enforce managed identity, private endpoint access, and ingress through Application Gateway and
  APIM rather than exposing workloads directly.
- Enable Azure Policy for AKS, Defender for Containers, cluster autoscaler, workload probes, and
  container image governance as Step 4 and Step 5 implementation baselines.
- Preserve the option to attach Azure Functions later for event-driven tasks that do not justify
  long-running pods.
