## 🔍 Context

The RFQ requires EU-only processing, replication, indexing, and caching unless Contoso provides a
written exception. The Step 2 assessment identified three specific sovereignty risks:

- Azure Front Door is excluded from the Azure EU Data Boundary.
- Entra External ID MFA methods that depend on PSTN, SMS, or push notification flows may process
  data outside the EU boundary.
- Azure Resource Manager EU Data Boundary settings must be enabled on a new or empty tenant before
  subscription creation and cannot be retrofitted casually later.

That means the architecture needs a compliance strategy, not just a service selection.

## ✅ Decision

Adopt an EU Data Boundary baseline with three rules:

1. Treat regional ingress through Azure Application Gateway and WAF as the default compliant edge
   path for the core platform.
2. Restrict Entra External ID MFA to FIDO2, passkeys, or TOTP-based methods unless Contoso grants
   a written exception for non-EU-sovereign MFA channels.
3. Require tenant-level EU Data Boundary configuration before creating the project subscriptions.

Azure Front Door may only be reintroduced through a formally documented exception path that records
the compliance trade-off and receives written Contoso approval.

## 🔄 Alternatives Considered

- `EU boundary baseline with App Gateway, MFA restrictions, and tenant prerequisite`
  Pros: best alignment with RFQ sovereignty requirements and lowest compliance ambiguity.
  Cons: loses some global edge capability and adds governance constraints up front.
  WAF impact: Security `+`, Compliance `+`, Performance `-`, Operations `-`.
- `Retain Front Door with written exception`
  Pros: preserves CDN and global edge acceleration.
  Cons: depends on customer exception handling and keeps a known compliance deviation in
  the design.
  WAF impact: Performance `+`, Security `-`, Compliance `-`.
- `Allow all Entra MFA methods`
  Pros: simplifies user experience and avoids MFA channel constraints.
  Cons: leaves a known residual data-boundary gap for PSTN and push-based authentication
  methods.
  WAF impact: Operations `+`, Compliance `-`.
- `Use an existing tenant without EU boundary enablement`
  Pros: fastest path if the tenant already exists.
  Cons: creates a foundational sovereignty risk that may be impossible to correct cleanly
  later.
  WAF impact: Schedule `+`, Compliance `-`.

## ⚖️ Consequences

### Positive

- Aligns the design to the RFQ's strongest compliance requirement before implementation begins.
- Removes ambiguity around Front Door by making the compliant default explicit.
- Creates a clear identity control baseline for Entra External ID MFA.
- Forces the tenant prerequisite to be handled early, when it is still cheap to correct.

### Negative

- Gives up Front Door's global CDN and anycast benefits unless a formal exception is approved.
- Narrows MFA choices for end users and administrators.
- Adds governance work and decision gates before subscription creation and edge-service rollout.

### Neutral

- A later business case can still justify Front Door, but only through a documented exception and
  compensating controls.
- The decision does not eliminate the need for ongoing reviews of Microsoft's published EU Data
  Boundary posture for Entra and edge services.

## 🏛️ WAF Pillar Analysis

- Security: `↑` Reduces sovereignty ambiguity and constrains identity flows to lower-risk
  MFA methods.
- Reliability: `→` Remains acceptable, but regional ingress removes Front Door's global
  edge failover posture.
- Performance: `↓` Regional ingress may not match Front Door for global edge acceleration
  and caching.
- Cost: `↑` Replacing Front Door with Application Gateway can reduce cost while adding
  governance overhead.
- Operations: `↓` Requires tighter tenant governance, MFA policy management, and explicit
  exception handling.

## 🔒 Compliance Considerations

- Directly addresses GDPR-aligned EU data residency requirements from the RFQ.
- Supports PCI-DSS segmentation by keeping ingress regional and identity controls explicit.
- Requires formal governance records for any approved exception involving Front Door or non-EU MFA
  channels.
- Must be paired with regional storage redundancy choices such as ZRS and private connectivity to
  data services.

## 📝 Implementation Notes

- Build the baseline production path around Application Gateway WAF, APIM, AKS, and private data
  services in `swedencentral`.
- Configure Entra External ID policies to allow FIDO2, passkeys, and TOTP while disabling SMS and
  push-based MFA unless explicitly approved.
- Confirm EU Data Boundary tenant readiness before provisioning subscriptions, policies, and
  workloads.
- Keep a formal architecture exception register for any requested use of Front Door.
