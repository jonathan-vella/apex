## 🔍 Context

The `my-demo` project is a Star Wars themed non-production web demo with a strict monthly target of about `$9` and a soft ceiling of `$15`. The architecture assessment identifies Azure Static Web Apps as the frontend platform and explicitly notes known IaC deployment reliability issues for the Free SKU.

The workload requires:

- public HTTPS access with low operational overhead
- predictable deployment through Bicep/ARM
- integration with Azure Functions and Application Insights
- best-effort SLA suitable for a demo

## ✅ Decision

Use **Azure Static Web App Standard SKU** in `westeurope` as the hosting tier for the demo frontend.

This preserves low cost while reducing deployment risk and enabling reliable automation in the 7-step workflow.

## 🔄 Alternatives Considered

| Option                             | Pros                                                                                | Cons                                                                            | WAF Impact                              |
| ---------------------------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- | --------------------------------------- |
| Static Web App Standard (selected) | Reliable ARM/Bicep deployment, built-in SLA, supports required integration patterns | Fixed monthly charge (`$9`)                                                     | Reliability: +, Cost: -, Operations: +  |
| Static Web App Free                | Zero direct hosting cost                                                            | Known deployability issues for ARM/Bicep, lower operational predictability      | Cost: +, Reliability: -, Operations: -  |
| App Service (Basic+)               | Flexible runtime model and broad hosting controls                                   | Significantly higher monthly cost for this demo profile, unnecessary complexity | Reliability: +, Cost: --, Operations: - |

## ⚖️ Consequences

### Positive

- Increases deployment consistency for Infrastructure as Code execution.
- Keeps architecture within the approved budget target.
- Supports clean integration with Functions and Application Insights.
- Reduces time spent on non-deterministic deployment troubleshooting.

### Negative

- Introduces a recurring base cost versus Free SKU.
- Provides capabilities beyond immediate demo needs.

### Neutral

- Does not change the public-access model or compliance scope.
- Does not materially alter the best-effort support approach.

## 🏛️ WAF Pillar Analysis

| Pillar      | Impact | Notes                                                                           |
| ----------- | ------ | ------------------------------------------------------------------------------- |
| Security    | →      | Public demo posture remains unchanged; HTTPS and platform defaults still apply. |
| Reliability | ↑      | More predictable deploy behavior than Free SKU for Bicep-driven workflows.      |
| Performance | →      | Similar static delivery path for this low-traffic profile.                      |
| Cost        | ↓      | Slightly higher than Free, but still inside the approved budget envelope.       |
| Operations  | ↑      | Simplifies repeatable deployments and lowers troubleshooting friction.          |

## 🔒 Compliance Considerations

- No additional compliance obligations are introduced.
- The workload remains non-production with no regulated data.
- Regional deployment remains in `westeurope`.

## 📝 Implementation Notes

- Provision Static Web App using Standard SKU in `westeurope`.
- Keep Functions on Consumption and Application Insights on Free tier.
- Maintain Managed Identity flow from Functions to Application Insights.
- Retain public access and HTTPS-only behavior for demo users.
